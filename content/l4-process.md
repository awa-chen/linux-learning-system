# 进程与信号

## 讲解

每一条运行的命令、每一个后台服务，在 Linux 里都是一个**进程（process）**，内核给它分配一个唯一编号 PID。进程不是孤立的：启动它的进程是它的"父进程"（PPID），一层层往上就是 init/systemd（PID 1）。理解进程树，是排查"谁卡住了、谁占满了 CPU"的基础。

**信号（signal）**是进程间最轻量的沟通方式：你没法直接让别人的进程改代码，但可以向它"发个信号"。`SIGTERM(15)` 是礼貌请求退出，`SIGKILL(9)` 是内核强行杀死（不对进程做任何清理），`SIGHUP(1)` 常用于让服务重读配置。默认情况下 `kill` 发的是 `SIGTERM`——能用 15 就别用 9，给程序一个优雅收场的机会。

> 本课可在交互沙盒中练习 `ps` 与管道；`top`/`kill` 等实时/写操作请在真实终端体验。

## 动手实验

1. 查看当前会话的进程（沙盒可练）：
   ```bash
   ps
   ```
   预期：列出当前 shell 及其子进程，含 PID、TTY、CMD 三列。

2. 以"全格式"查看所有进程：
   ```bash
   ps -ef
   ```
   预期：多出 UID、PPID 列，`-e` 表示所有进程、`-f` 表示完整格式。

3. 用管道过滤出 bash 相关进程（沙盒可练）：
   ```bash
   ps | grep bash
   ```
   预期：只显示命令行里包含 `bash` 的行，管道把左边输出喂给右边 `grep`。

4. 实时观察（真实终端）：
   ```bash
   top
   ```
   说明：进入后按 `P` 按 CPU 排序、`M` 按内存排序、`q` 退出。

5. 优雅结束一个进程（真实终端）：
   ```bash
   kill 1234          # 发 SIGTERM(15)
   kill -9 1234       # 仍不死才用 SIGKILL
   ```
   说明：先 15 后 9，是运维的礼貌与底线。

## 常见错误

- **`kill` 提示 "No such process"**：PID 已经结束，或你不是该进程 owner/root。先 `ps` 确认 PID 还活着。
- **`kill -9` 当成首选**：SIGKILL 不给进程清理机会，可能造成数据未落盘、锁文件残留。务必先试默认 15。
- **分不清 PID 和 PPID**：想杀的"父服务"其实是 PPID，杀错子进程会被父进程立刻重启。看清 `ps -ef` 的父子列。
- **`top` 里乱按以为卡死**：其实在跑，按 `q` 即可退出；若误按了 `k` 会进入杀进程流程，按 `Esc` 取消。

## 小结

1. 进程有 PID/PPID，形成进程树；`ps` 看快照，`top` 看实时。
2. 信号是控制进程的"按钮"：默认 `SIGTERM(15)` 礼貌，`SIGKILL(9)` 最后手段。
3. 管道 `|` 能把 `ps` 的输出交给 `grep` 过滤，是日常排查的标配组合。

进阶：下一课 `systemd` 会告诉你，现代 Linux 里这些进程大多由 `systemctl` 统一管理，而不是直接手敲 `kill`。
